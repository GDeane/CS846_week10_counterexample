**Why this is sub-optimal:**

The guideline forces a single category onto each finding, but all three findings here
describe the *same underlying defect* from three different angles:

- Is `parse_date` returning `None` a **Bug Fix** problem? Only if you treat the docstring
  as the contract — then the code violates its stated interface and `days_until` crashes.
- Is it a **Documentation** problem? Only if you treat the silent-`None` behaviour as
  intentional — then the docstring simply needs to catch up.
- Is it an **Enhancement**? If the real goal is a structural redesign (raise
  `ValueError`, add an `Optional[datetime]` type hint, use a `Result` type), that is a
  correctness-by-design improvement, not a bug patch or a docs tweak.

Because the guideline requires *exactly one* label per finding, Copilot split the defect
into three separate findings and assigned each a different priority: P1 (crash), P2
(improvement), P3 (docs). A developer reading only the P1 item will add a `None`-check in
`days_until` without ever touching the docstring or the `None`-return design — leaving two
of the three findings unresolved. A developer reading only the P3 item will update the
docstring and close the ticket, never realising the crash still exists.

The root problem is that the correct fix depends on a design decision: is `None` the
intended return value or not? If yes, the docstring and every caller must be updated. If
no, the return value must change and the docstring becomes correct. The taxonomy cannot
encode this dependency. Forcing a label before that decision is made gives the appearance
of triage without the substance of it, and the split-priority output actively encourages
an incomplete fix.

### 3. New and Updated Guidelines that worked for Problem A_1

#### Guideline 3 (Revised): Flag Design Decision Issues and Escalate

**Description:**

When a finding genuinely spans more than one category (BUG FIX, ENHANCEMENT, DOCUMENTATION) — because the right fix
depends on which artefact (code or contract) is treated as the source of truth — do not force a single label. Instead:

1. Mark the finding as **Design Decision** and list every category it touches.
2. State the competing interpretations explicitly (e.g., "if the `None` return is
   intentional, update the docs and all callers; if it is a bug, change the return value").
3. Escalate the design decision to the code author before suggesting a concrete fix.
4. Only after the design decision is recorded, apply the single-category taxonomy and
   priority to the *resolved* sub-findings.

**Reasoning:**

The single-label taxonomy is effective for genuinely independent findings (a standalone
off-by-one, a missing docstring on an unrelated function). It breaks down when the
category itself is the thing under debate, because premature labelling misdirects the fix
rather than clarifying it. Recognizing cross-cutting issues and surfacing the design
question first prevents the incomplete-fix failure mode described before.

Context Given: `parse_date.py`

    > decision (e.g., which artifact is the source of truth), mark it **Cross-Cutting**,
    > list the competing interpretations, and flag it for the code author to resolve before
    > a fix is applied.

### New and Updated Guidelines That Worked for Problem C_1

### Revised Guidelines

To improve the review quality, the guidelines were refined in the following way:

- Guideline 1 should require that the reviewer explicitly understand and state the intended behavior of the code before
  evaluating correctness.
- Guideline 4 should require the reviewer to assess whether the proposed change introduces regression risk, especially
  in security-sensitive logic such as authentication or access control.

Providing this context helps the model analyze the code more accurately.

**Description**

The reviewer must understand the intended behavior of the function before evaluating whether the code is correct.
In addition, the reviewer must assess whether the proposed change introduces regression risk, particularly in
security-critical logic such as authentication or access control.

**Reasoning**

Providing the intent of the function helps the model understand how the system is expected to behave.
Requesting regression-risk analysis ensures that changes are evaluated not only for correctness but also for potential
side effects that could weaken security.

**Improved Prompt:**

Intent:

If token validation fails or throws an exception, access must be denied.

For each finding provide:

At the end provide a final verdict:
